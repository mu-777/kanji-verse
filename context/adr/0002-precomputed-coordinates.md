# ADR-0002: 2D座標の事前計算方式

## ステータス
accepted

## 日付
2026-03-16

## コンテキスト
- 3000ノードを夜空風にレイアウトする必要がある
- 静的ホスティング（GitHub Pages）
- ユーザの要求: インタラクティブなアプリにしたい

## 検討した選択肢

### 選択肢A: 2D座標を事前計算してJSONに焼き込む（採用）
- メリット: ブラウザ負荷ゼロ。レイアウトが安定・再現可能。高速起動。
- デメリット: レイアウト変更には再計算が必要

### 選択肢B: Embeddingベクトルをブラウザに渡し、umap-jsで計算
- メリット: パラメータ変更によるインタラクティブな再計算が可能
- デメリット: 初回ロードに20〜60秒かかる。UMAPは確率的なので結果が毎回変わり「この漢字はここにある」という安定感がない。ベクトルJSONが ~11MBになる。

## 決定
2D座標を事前計算してJSONに焼き込む方式を採用する。

## 理由
ユーザの「インタラクティブ」の意図を確認したところ、UXのインタラクティブさ（パン・ズーム・検索・クリック）を求めており、レイアウトのリアルタイム計算を求めていなかった。
安定したレイアウト、高速な起動、GitHub Pagesへの適合性を重視した。

## 影響
- Python スクリプトでUMAP計算 + public/data/kanji.json 生成が必要
- データ更新時は再実行が必要

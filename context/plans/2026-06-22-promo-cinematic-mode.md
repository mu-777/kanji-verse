## アプローチの選択肢

### 選択肢A: アプリ内スクリプト型シネマティックモード（URLトリガ）＋ OBS実時間キャプチャ【推奨】
`?demo=1` で起動すると、chrome を隠し、既存API（intro/flyTo/searchByMeaning/selectNode）を時間軸で順に駆動する「決め打ちの演出タイムライン」を再生。`&loop=1` でループ。OBS でブラウザを録画。
- メリット: 決定論的で何度でも同条件で撮り直せる。操作ブレがない。タイミングを秒単位で調整可。**ライブサイトの「アトラクトモード」としても再利用可**。実装が中規模。
- デメリット: 実時間キャプチャのため、UHD 620＋Bloom でフレーム落ちの可能性（→ 後述の保険あり）。

### 選択肢B: 手動操作を撮るだけ（コードなし）＋ ショットリスト
- メリット: 実装ゼロ。
- デメリット: ドラッグ操作がブレる、"love"→愛 の timing を毎回綺麗に決められない、詳細パネルの出る位置・タイミングが安定しない。**この製品の売り＝滑らかな運動**なので相性が最悪。

### 選択肢C: ヘッドレス固定タイムステップでフレーム連番出力 → ffmpeg エンコード
Puppeteer で固定 dt 駆動、canvas を毎フレーム PNG 化、ffmpeg で結合。
- メリット: 機械が遅くても完璧に滑らか、4K/60fps も自在、完全再現。
- デメリット: 実装が最重量。ヘッドレスGL での Bloom/postprocessing 描画が不安定（WSL2＋統合GPU）。レンダループに固定時計フックが必要。

## 選択したアプローチ
**選択肢A を採用**。理由: ユーザが「シネマティックモード実装」を選択済み。効果（再現性・品質）/工数のバランスが最良で、副産物としてライブサイトのアトラクトモードにもなる。
- **保険**: A の実時間キャプチャが UHD 620 で滑らかでない場合に備え、タイムラインを「固定 dt でも駆動できる」設計（dt を外部注入可能）にしておき、必要時に C へ最小コストで発展できるようにする。

## 演出タイムライン（master ≈ 28s / X用に 0–15s を切り出してループ）
1. **0–4s ヒーロー**: chrome 非表示、銀河全体を中距離からゆっくり自動回転（掴みの一枚画）。
2. **4–9s 意味検索の発光**: 検索ボックスに `love`（テーマ語は調整）をプログラムでタイプ → 一致漢字が緑に発光。ゆっくりオービットして分布を見せる。※「意味が近い＝近くに集まる」が伝わる語を tuning で選定。
3. **9–16s 1字へダイブ**: クリア → `愛` へ `flyTo`、寄りでラベル出現、詳細パネルを開き 読み(ローマ字)＋意味 を表示。
4. **16–24s 引き**: パネルを閉じ、広角ポーズへ pull-back → 全体像へ。ゆっくり回転。
5. **24–28s ループ継ぎ目**: 冒頭1と画作りを合わせ、`loop=1` でシームレスに 1 へ戻る。

テロップ/タイトルは編集段（ffmpeg）で付与（アプリ内には焼かない）。

## 検証方法
- `npm run build` → `npx vite preview` → `…/kanji-verse/?demo=1&loop=1` で再生。Windows ブラウザで以下を確認:
  - 各ビートが設計どおりの順・尺で再生される（welcome が出ない、chrome 制御が正しい）。
  - `?fmt=square` 等でウィンドウ比率を変えても破綻しない（responsive）。
  - ループ継ぎ目が許容できる滑らかさ。
  - 通常起動（`?demo` なし）が**従来どおり**＝デグレなし。
- 実機 UHD 620 でキャプチャ相当の負荷時のフレーム感を体感確認。滑らかでなければ保険（C）への発展を相談。
